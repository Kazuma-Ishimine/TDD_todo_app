## OrchestratorAgent のルールを強化した

### 背景

今日の TDD セッションで、OrchestratorAgent がサブエージェント（Red / Green）のファイル作成に失敗した際に、**自分でコードを書いてしまう**という問題が発生した。

- Red Agent が `app.test.ts` を作らなかった → OrchestratorAgent が general-purpose エージェント経由でテストを作成
- Green Agent が `index.ts` を実装しなかった → 同様に自前で実装

これは「オーケストレーターは指揮者であり、自分でコードを書いてはいけない」というルールに反する動作だった。

### 変更内容

`/.github/agents/OrchestratorAgent.agent.md` を v2.0.0 → **v2.1.0** に更新した。

**Prohibited Actions の強化**

```
❌ Generate code yourself — ABSOLUTE.
   Never use create/edit tools to write source code, test code, or configuration files.
   This includes recovery attempts when a sub-agent fails.

❌ Fall back to writing code when a sub-agent fails —
   retry the agent or stop and ask the user for guidance instead
```

**Workflow フェーズへの明示的な失敗時対応の追加**

- Red / Green フェーズ：エージェントがファイルを作成しなかった場合は「1回だけリトライし、それでも失敗したらユーザーに報告して止まること」を明記
- Refactor フェーズ：「変更なし」も正当な結果として受け入れること（自分でリファクタしない）

**Key Principles の強化**

```
❌ Write code — under any circumstances, including fallback
❌ Create or edit source files directly
❌ Compensate for a failing sub-agent by doing the work yourself
```

### 教訓

サブエージェントが失敗したとき、オーケストレーターが「穴を埋めるために自分でやる」のは一見合理的に見えるが、役割の分離を破壊する。失敗を見えなくするより、**失敗を報告してユーザーに判断を委ねる**方が正しい設計。

### Red フェーズ：テストを先に書く

`backend/src/app.test.ts` を新規作成した。

Hono の `app.request()` を使えばサーバーを起動せずに HTTP レベルのテストが書けるので、それを活用した。  
各テストが独立して動くように、`clearStorage()` を実装側からエクスポートして `beforeEach` でストアをリセットする設計にした。

カバレッジの設計方針：

- 正常系（201 / 200 のレスポンス形状チェック）
- バリデーションエラー（422：欠如・空文字・空白のみ・文字数超過）
- 一意制約違反（409：重複 name）
- 存在しないリソース（404）
- ソフトデリート後に一覧・単体取得ともに見えなくなること
- App 削除時に配下の Todo もカスケード削除されること
- 境界値（name が100文字 → OK、101文字 → 422）

この段階では実装がないため全テストが失敗する（Red）。

### Green フェーズ：テストを通す実装

`backend/src/index.ts` を全面的に書き直した。

**設計のポイント：**

- ストレージは `Map<string, AppRecord>` / `Map<string, TodoRecord>` のインメモリ実装（DB 接続なし）
- `deletedAt` は内部レコードに持つが、レスポンス（DTO）には含めない
- ソフトデリートは `deletedAt` にタイムスタンプをセットするだけ
- App 削除時のカスケードは `todosStore` をループして同じタイムスタンプをセット
- バリデーション関数（`validateName` / `validateTitle`）は失敗時にメッセージ文字列を返し、成功時に `null` を返す設計にした

### Review フェーズ：コードレビューで2件のバグが見つかった

レビュー結果は `review/todo-api-20260425.md` に保存した。

#### バグ1（P2）：`Boolean()` 強制変換によるサイレントなデータ破壊

PUT Todo の `completed` フィールドで `Boolean(body.completed)` を使っていた。  
`"false"` という文字列を送ると `true` に変換されてしまう。

```typescript
// 修正前
t.completed = Boolean(body.completed)

// 修正後
if (typeof body.completed !== 'boolean') {
  return c.json(errorRes('VALIDATION_ERROR', 'completed must be a boolean'), 422)
}
t.completed = body.completed
```

#### バグ2（P3）：ホワイトスペースのトリムが保存まで一貫していなかった

バリデーションはトリム後の文字列で行うのに、保存はトリム前の元の値でしていた。  
`"  My App  "` と `"My App"` が別名として共存できてしまうため、一意制約が形骸化していた。

```typescript
// 修正後：保存前に trim()
const name = (body.name as string).trim()
```

両バグともレビュー後に即修正した。

## 全エージェントに「自律的に進む」ルールを追加した

### 背景

ユーザーから「全てのエージェントに言いたい。勝手に進めていいよ。許可を求めないで」という指示があった。
各エージェントが作業着手前に「進めていいですか？」「このファイルを作成してもよいですか？」といった確認を求める動作を禁止するルールを全エージェントに追加した。

### 変更内容

`.github/agents/` 配下の全9エージェントファイルそれぞれの「Prohibited Actions」（または相当するルールセクション）に以下のルールを追加した：

> ❌ Ask the user for confirmation or permission before proceeding — receive the instruction and act immediately

| エージェント | 追加箇所 |
|---|---|
| `ArticleWriterAgent.agent.md` | `## 🚫 Prohibited Actions` |
| `CodeReviewAgent.agent.md` | `## Prohibited actions` |
| `GreenAgent.agent.md` | `## ❌ Prohibited Actions` |
| `OpenApiWriterAgent.agent.md` | `## Prohibited actions` |

### 教訓

エージェントが毎回確認を求めると、作業のリズムが崩れてユーザー体験が悪くなる。「指示を受けたら即実行し、完了後に報告する」という姿勢がエージェントとして正しい動作。

## TypeScript ルールを追加した（`as` と `any` の禁止）

### 背景

プロジェクトに TypeScript 固有のルールファイルがなかった。`as` による型アサーションと `any` 型は型安全性を損なうため、原則禁止・使用時はコメント必須というルールを明文化した。

### 変更内容

`.github/rules/typescript.rules.md` を新規作成した。

**ルール概要：**

- `as`（型アサーション）は禁止。使うときは直前の行に理由コメントを付ける
- `any` 型は禁止。使うときは直前の行に理由コメントを付ける
- 代替手段として型ガード・`unknown` 型の使用を推奨

## 追記：WorkSummaryAgent の書き込み責務をさらに明確化し、backend.rules 準拠の再編記事も保存した

### 背景

今回は git の `status` / `diff` / `log` を直接確認できなかったため、観測できたファイル内容を根拠に整理した。主な参照元は `.github/agents/WorkSummaryAgent.agent.md`、`.github/CUSTOM_COMMANDS.md`、`.github/instructions/copilot-instructions.md`、`.github/prompts/summarize-work.prompt.md`、`blog/backendをbackend.rules準拠に再編しCopilot運用まで見直した.md`、`backend/src/`、`.github/rules/backend.rules.md`。

### やったこと1：`WorkSummaryAgent` を「返答するだけで終わらない」前提に寄せた

`WorkSummaryAgent` まわりの定義を読むと、日報生成の責務がかなり明確になっていた。

- `.github/agents/WorkSummaryAgent.agent.md` に、`diary/YYYYMMDD.md` へ**直接書き込み**、同日ファイルがあれば**追記**、未完了事項も書く、という完了条件が入っている
- `.github/CUSTOM_COMMANDS.md` でも、Copilot CLI では `@WorkSummaryAgent` が `diary\YYYYMMDD.md` に書き込み、既存ファイルには append する動作として案内されている
- `.github/instructions/copilot-instructions.md` では、実装や修正の完了後に `@ArticleWriterAgent` → `@WorkSummaryAgent` を自動で回す流れが整理されている
- `.github/prompts/summarize-work.prompt.md` は軽量な要約用として残っているが、ファイル出力の責務は agent 側に寄せる構成になっている

要するに、`WorkSummaryAgent` は「本文を返すだけの要約係」ではなく、`diary/` を更新する実務フローの一部として固定されつつある。

### やったこと2：backend.rules 準拠で再編されたバックエンドと周辺運用を記事化した

新しく `blog/backendをbackend.rules準拠に再編しCopilot運用まで見直した.md` が保存されており、バックエンド実装そのものだけでなく、周辺の開発運用まで含めて整理していた。

記事内で確認できた主なポイントは次の通り。

- `.github/rules/backend.rules.md` を基準に、`backend/src/` を `domain/` `usecase/` `interface/` `infrastructure/` に分ける構成へ寄せている
- `backend/src/index.ts` は registry 経由の薄い入口になっており、責務を各層へ分配している
- `backend/src/interface/controllers/request-validation.ts` に入力検証を集め、`Boolean()` のような黙った型変換を境界で止める形にしている
- `backend/src/interface/presenters/http-presenter.ts` で DTO 整形とエラーの HTTP 変換を受け持たせている
- `backend/package.json`、`.github/workflows/backend.yaml`、`.github/workflows/frontend.yaml` から、lint / typecheck / test と PR コメント連携まで整えていることが読める
- `.github/CUSTOM_COMMANDS.md` や各 agent 定義も含め、実装 → レビュー → PR → 記事 → 日報までをひとつの流れとして扱っている
