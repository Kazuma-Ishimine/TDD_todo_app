## マイグレーション 003 を idempotent に修正した

### 背景

コードレビュー（`review/feat-mysql-integration-tests-ci-2026-04-25.md`）で P1 指摘があった。  
`003_drop_app_name_unique_index.sql` は当初 `ALTER TABLE App DROP INDEX idx_app_name;` の1行だけだった。  
新規環境や CI の fresh データベースでは `idx_app_name` が存在しないため、マイグレーションがエラーで停止していた。

### 問題の核心

MySQL 8.0 は `DROP INDEX IF EXISTS` 構文をサポートしていない。  
PostgreSQL や MariaDB では使える構文だが、MySQL 8.0 では構文エラーになる。

### 対応内容（コミット `8dfc35d`）

`information_schema.STATISTICS` でインデックスの存在を確認し、Prepared Statement で実行する SQL を切り替えるパターンに書き換えた。

```sql
SET @exist := (
  SELECT COUNT(*) FROM information_schema.STATISTICS
  WHERE table_schema = DATABASE()
    AND table_name   = 'App'
    AND index_name   = 'idx_app_name'
);
SET @sql := IF(@exist > 0, 'ALTER TABLE App DROP INDEX idx_app_name', 'SELECT 1');
PREPARE stmt FROM @sql;
EXECUTE stmt;
DEALLOCATE PREPARE stmt;
```

インデックスが存在しない新規環境では `SELECT 1` が実行されるだけなので、マイグレーションがブロックされない。  
接続設定の `multipleStatements: true` が前提となるが、マイグレーション専用接続にはすでに設定済みだった。

同コミットで `review/feat-mysql-integration-tests-ci-2026-04-25.md` にも「fixed」の返信コメントを追記した。

### 残課題

同じレビューで指摘された `existsActiveByName()` と `save()` の間の race condition については、`AppRepository` インターフェースにトランザクション境界を導入する大きな変更が必要なため、本 PR スコープ外として既知の制限事項（known limitation）とした。別 Issue での対応を推奨。

## テストコードの `as` 型アサーション違反を修正した

### 背景

前セッションのコードレビュー対応で統合テストを整備した際、  
`as Record<string, unknown>` が3つのテストファイルに紛れ込んでいた。  
このリポジトリには `typescript.rules.md` により「`as` は禁止。必要な場合は直前に理由コメントを必須とする」ルールがある。

### 対応内容（コミット `50d59cb`）

#### テストコード：不要なキャストを削除

`toHaveProperty()` は `unknown` 型を受け付けるため、`as Record<string, unknown>` でキャストする必要がない。  
型情報を捨てるだけで型安全性に何も貢献していないため、3ファイルのキャストを削除した。

| ファイル | 変更 |
|---|---|
| `backend/src/tests/integrations/controllers/http-presenter.test.ts` | `as Record<string,unknown>` を2箇所削除 |
| `backend/src/tests/integrations/controllers/app-controller.test.ts` | `as Record<string,unknown>` を1箇所削除 |
| `backend/src/tests/integrations/controllers/todo-controller.test.ts` | `as Record<string,unknown>` を1箇所削除 |

#### 本番コード：理由コメントを追加

`request-validation.ts` の `toRecord()` 関数には `as Record<string, unknown>` が残っているが、  
`typeof value !== 'object'`・`value === null`・`Array.isArray(value)` の3つのガード通過後なので `as` キャストは正当だ。  
ルール準拠のため、直前に理由を説明するコメントを追加した。

```typescript
// value has been narrowed to a non-null, non-array object by the guards above; cast to Record is safe
return value as Record<string, unknown>;
```

### ポイント

テストコードも本番コードと同じ規約水準で管理することで、「なぜこの `as` があるのか」という疑問が生まれないテストスイートを維持できる。

**関連ファイル**
- `backend/src/controllers/request-validation.ts`
- `backend/src/tests/integrations/controllers/http-presenter.test.ts`
- `backend/src/tests/integrations/controllers/app-controller.test.ts`
- `backend/src/tests/integrations/controllers/todo-controller.test.ts`
